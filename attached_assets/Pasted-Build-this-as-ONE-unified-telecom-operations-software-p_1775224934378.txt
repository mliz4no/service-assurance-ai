Build this as ONE unified telecom operations software platform inside a single Replit project.

This is not a separate quoting app. It is one modular platform with multiple modules that share the same core data model and lifecycle.

Modules:
- Quoting / Sales
- Contracting
- Service Order / Technical Order Forms
- Service Delivery
- Service Assurance AI

Architecture requirements:
- Use a modular monolith architecture inside one codebase.
- Use a shared core data model for Customer, Location, Service, Circuit, Contact, Escalation Rules, SLA Profile, Devices/Handoffs, Contracts, Orders, and Assurance objects.
- Do not duplicate customer or service records between modules.
- Each module should have clear service boundaries, but all should operate on the same platform data model.
- Use internal events/workflows so lifecycle changes in one module automatically update the others.

Critical lifecycle requirement:
- A customer should NOT appear as active inside Service Assurance AI during quoting or contract stage.
- Only when a Service Delivery Order reaches Completed or Activated status should the platform automatically create or update the corresponding Active Customer, Active Site, Active Service, Circuit/Link, Handoff, Monitoring Targets, Customer Internal Escalations, Vendor Escalations, and SLA/Severity settings inside the Service Assurance AI module.

Automatic handoff requirement:
- When Service Delivery Order status becomes Completed or Activated, automatically trigger an internal provisioning workflow that creates or updates the Assurance-side records.
- If the customer already exists, append new locations/services and avoid duplicates.
- Maintain full traceability from the Active Service back to Quote, Accepted Quote, Contract, Service Order Form, and Service Delivery Order.

UI requirement:
- Show end-to-end lineage on customer/service records:
  Quote -> Accepted Quote -> Contract -> Signed Service Order Form -> Service Delivery Order -> Active in Service Assurance

Technical requirements:
- Build this as a serious enterprise B2B app, not a demo.
- Use mocked external integrations first.
- Structure code so real integrations can be added later without major refactoring.
- Keep modules separate in code organization, but unified in data model and workflow.

Important naming:
- Use the term “Customer Internal Escalations” for customer-facing escalation paths.
- Keep “Vendor Escalations” separate from customer internal escalation logic.
- Use the term “Service Delivery Order” for the final order that implementation teams work from.
